2. HTTP Beacon 流量特征

2.1 心跳间隔（最基础特征）

  • 默认 Sleep = 60 秒（可在 CS 界面调整）
  • 规律性 HTTP 请求，间隔高度一致
  • sleep 0 模式：连续高频请求，无间隔

检测规则：同一 IP 对同一 C2 的 HTTP 请求间隔在 55-65 秒范围内（允许网络抖动），持续 3 次以上。

2.2 HTTP-GET 请求（元数据上报）

默认未修改 Malleable C2 时：

关键特征：

  • Cookie 字段包含 Base64 编码数据（元数据），通常以 MIG 开头（RSA 加密后）
  • 元数据大小：通常 128 字节（RSA-1024 加密后）
  • URI 路径较短

2.3 HTTP-POST 请求（结果回传）

关键特征：

  • /submit.php?id= 模式（默认配置残留，非常明显的指纹）
  • POST body 包含 AES 加密的执行结果
  • id 参数为递增数字

2.4 HTTP 响应特征

  • 任务响应内容类型为 application/octet-stream 或 text/plain
  • 无任务时返回 空 HTTP 200 或极短响应

2.5 默认 User-Agent（CS 4.0 之前）

CS 4.0 之前：每次请求的 User-Agent 完全相同，不随浏览器版本变化

  • 这是最粗暴的检测特征

CS 4.0+：内置随机 UA 生成，但仍存在规律：

  • UA 版本号过旧（如 Windows NT 6.1 等已淘汰版本）
  • UA 与实际目标 OS 版本不匹配

3. HTTPS Beacon 流量特征

3.1 TLS 证书特征

默认证书（高价值 IOC）

CS Team Server 的默认 TLS 证书：

字段 默认值
Issuer CN Major Cobalt Strike
Subject CN Cobalt Strike
Organization cobaltstrike
Serial Number 0x00 或固定值
有效期 默认起始日期（如 2014/01/01 等老日期）

几乎所有改版/破解版仍然保留默认证书指纹。

已知默认证书 SHA-256 指纹

3.2 JA3/JA3S 指纹

JA3（客户端 TLS 指纹）：

  • 默认 CS Beacon 的 JA3 指纹是固定的
  • 常见值：72a589da586844d7f0818ce684948eea
  • 由于 CS 使用 Java TLS 库（Bouncy Castle），JA3 与标准浏览器差异明显

JA3S（服务端 TLS 指纹）：

  • Team Server 的 JA3S 同样固定
  • 常见值：ae4edc6faf64d08308082ad26be60767

💡 检测策略：监控与非浏览器 JA3（如 72a589da...）的 HTTPS 连接，尤其是目标为非 CDN IP 的连接。

3.3 TLS 会话特征

  • TLS 版本：通常使用 TLS 1.2
  • 密码套件：较少（CS 不协商大量套件），与浏览器行为不同
  • 证书链：自签名，通常只有 1 层（无 CA 链）

4. DNS Beacon 流量特征

4.1 通信原理

Beacon 通过 DNS 查询将数据编码到子域名中传输：

4.2 DNS 查询特征

特征 描述
查询类型 主要使用 TXT 记录（默认），也用 A/AAAA
子域名长度 长且固定模式，编码后通常较长
查询频率 高频查询，间隔短（相比正常 DNS）
子域名模式 包含 beacon 或编码后的数据段
单次查询数据量 TXT 模式可携带更多数据

4.3 DNS 流量模式

检测要点：

  • 单个域名在短时间内被同一主机大量 TXT 查询
  • 子域名熵值异常高（编码导致）
  • 子域名长度远超正常域名
  • 查询的域名无正常域名的随机性分布模式

4.4 DNS Beacon 类型

类型 说明 隐蔽性
DNS TXT Beacon 通过 TXT 记录传输数据 中等
DNS A Beacon 通过 A 记录的 IP 编码数据（每次 4 字节） 较高
DNS AAAA Beacon 通过 IPv6 地址编码（每次 16 字节） 较高

5. SMB Beacon 流量特征

5.1 通信机制

  • 使用 Windows 命名管道（Named Pipe）进行通信
  • 父 Beacon 和子 Beacon 之间通过 SMB 协议交换数据
  • 无外网流量，所有通信在局域网内完成

5.2 命名管道特征

特征 默认值/模式
默认管道名 \\<host>\pipe\msagent_<hex>
管道名模式 msagent_*, postex_*, status_*
访问方式 SMB2 协议，IPC$ 共享

5.3 检测方法

网络层特征：

  • SMB2 连接频繁建立/断开
  • 目标主机间频繁的 IPC$ 访问
  • 异常的 SMB Named Pipe 读写模式

6. TCP Beacon 流量特征

6.1 特征

  • 纯 TCP 连接，不加密
  • 固定端口监听
  • 流量完全明文可见
  • 通常仅用于内网短距离通信

6.2 检测

  • 非标准端口上的长连接 TCP 会话
  • 直接分析数据包即可看到明文指令

8. TLS/SSL 指纹特征

8.1 JA3 指纹详解

JA3 = MD5(Version, Ciphers, Extensions, EllipticCurves, ECPointFormats)

CS 默认 JA3 计算中涉及的特征：

  • ClientHello 中的密码套件数量较少
  • 使用 Java 典型密码套件顺序
  • 扩展字段少于浏览器
  • 不支持 HTTP/2 (ALPN)

8.2 已知 CS JA3 指纹库

8.3 JA4/JA4+ 指纹

新一代指纹标准 JA4 提供更精确的 TLS 指纹识别，对 CS 的检测更有效。

8.4 证书指纹

Elastic、Sigma 等社区维护了大量已知 CS 证书指纹列表。

9. Malleable C2 Profile 与规避

9.1 什么是 Malleable C2

Malleable C2 允许操作员自定义 Beacon 的网络行为，使流量外观完全模仿合法网站/应用。

9.2 Profile 关键配置块

9.3 常见伪装 Profile

伪装目标 Profile 来源
jQuery / jQuery CDN cobalt-strike-profiles
Google Analytics 自定义
Microsoft Teams API 自定义
Amazon AWS API 自定义
Cloudflare 自定义

9.4 Profile 仍可检测的原因

即使使用 Malleable C2，以下特征仍然存在：

  1. TLS 指纹未变：Profile 只改 HTTP 层，不改 TLS 握手
  2. 证书问题：自签名或证书不匹配目标域名
  3. Jitter 不完美：随机抖动模式与真人行为仍有差异
  4. 心跳模式：即使加了 jitter，周期性仍然可被统计分析捕获
  5. 内容特征：加密数据的高熵值仍然异常
  6. DNS/IP 不匹配：请求的 IP 与 CDN 节点分布不匹配

11.3 解密方法

如果获取到 Team Server 的 RSA 私钥（泄露/破解版常见）：

  1. 从 Cookie 提取 Base64 元数据
  2. RSA 私钥解密 → 得到 16 字节随机数
  3. SHA-256 派生 AES Key 和 HMAC Key
  4. 用 AES Key 解密所有后续 HTTP-POST/HTTP-GET 数据

13. 常见规避手法与对抗策略

13.1 攻击侧规避手法

手法 实现方式 检测难点
Malleable C2 自定义 HTTP 流量外观 URI/Header 指纹失效
域前置 (Domain Fronting) 借用 CDN 域名隐藏真实 C2 目标 IP 不是 C2
重定向器 (Redirector) 使用多层跳板 需关联多跳流量
自签名合法证书 购买/申请合法证书 证书特征失效
DNS-over-HTTPS DNS 查询加密 DNS 流量不可见
修改 Sleep/Jitter 增加间隔随机性 时间模式变弱
Cloudflare Workers 利用 Serverless 中转 流量混入 CDN
DNS TXT → DNS A 减少 DNS 数据量 DNS 检测变难
