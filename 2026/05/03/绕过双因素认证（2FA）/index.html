绕过双因素认证（2FA）

在最近对一个金融科技 Web 应用进行的授权渗透测试中，我发现了一个严重漏洞，使我能够完全绕过敏感账户操作上的双因素认证（2FA）——具体来说是密码更改和账户删除。这个发现之所以特别严重，是因为它的简单性：服务器实际上从未验证请求中是否存在 OTP。

目标

详情
类型 金融科技 Web 应用（支付/钱包平台）
认证流程 邮箱 + 密码 → 2FA OTP（通过短信/邮件发送） → 执行操作
范围 完整的已认证用户测试

应用在敏感操作上强制执行 2FA。在执行密码更改或账户删除之前，系统会提示用户输入 6 位 OTP。表面上看起来很安全。

发现过程

在使用 Burp Suite 映射应用功能时，我注意到了一个有趣的现象——OTP 作为明文参数出现在 POST 请求体中。我的第一个想法是：如果我直接……把它删掉会怎样？

核心问题是服务端缺少验证。OTP 检查仅存在于前端 JavaScript 中。后端 API 假设如果请求到达了它，前端已经验证了 OTP——这是一个经典且危险的假设。

影响

此漏洞被评为严重级别，原因如下：

影响 说明
账户接管 攻击者使用窃取的凭据可以在没有受害者手机的情况下更改密码
不可逆的账户丢失 任何已认证会话都可以删除账户，绕过 2FA
2FA 完全失效 整个第二因素在这些端点上提供零保护
财务风险 在金融科技平台上，账户接管直接威胁资金安全

CVSS 评分：~9.1 严重（AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H）

缓解措施

1. 将 OTP 强制为服务端必需字段

后端必须拒绝任何 otp_code 缺失、为空或为空值的请求——永远不要将其视为可选。

2. 先验证再执行

OTP 验证必须在任何敏感操作执行之前进行。如果验证失败或被跳过，绝不继续。

3. 永远不要信任前端

所有安全控制必须存在于服务器上。前端验证是 UX 便利，不是安全边界。

4. OTP 过期和一次性使用强制

每个 OTP 应在 5-10 分钟后过期，并在一次成功使用后立即失效。

5. 速率限制和异常检测

标记和阻止没有有效 OTP 的对敏感端点的重复请求。

6. 安全代码审计

审计所有敏感端点的条件验证 vs 强制验证模式。
