缓存投毒 - 2

缓存投毒 - 2

一. XSS

1：通过缓存投毒的反射型 XSS

赏金范围：$500 — $1,500
影响：低（需要受害者访问特定 URL）

局限性：

  • 受害者必须访问完全相同的投毒 URL
  • 缓存可能在受害者访问前过期
  • 看起来像典型的反射型 XSS

2：通过缓存投毒的存储型 XSS

赏金范围：$2,000 — $5,000
影响：中（自动影响多个用户）

生成的响应：

优势：

  • 无需用户交互
  • 影响热门页面的所有访客
  • 展示真实影响

真实案例：TikTok 通过缓存投毒实现存储型 XSS — 研究者通过投毒缓存绕过了 URL 编码限制，将反射型 XSS 转变为跨所有现代浏览器的存储型 XSS。

3：通过 Cookie 窃取实现账户接管

赏金范围：$5,000 — $12,000
影响：高（大规模凭据盗窃）

要求：

  • Session Cookie 没有 HttpOnly 标志
  • 没有 CSP 阻止外部请求
  • 高流量的热门页面

真实案例：Expedia 缓存投毒导致账户接管（报告 #1760213） — hav Cookie 被反射到响应中，允许 XSS 窃取 Session Cookie。缓存投毒将其变为影响该端点所有用户的存储型 XSS。

升级提示：计算受影响用户并展示大规模影响：

“该端点每天收到 50,000 次访问。在 30 秒的缓存 TTL 期间，估计有 416 个用户的会话会被窃取。”

4：OAuth 流程劫持

赏金范围：$10,000 — $25,000
影响：关键（完整的账户接管链）

完整的 OAuth 劫持链：

步骤 1：投毒主页

步骤 2：攻击者的 oauth-hijack.js

步骤 3：投毒 OAuth 回调处理器

步骤 4：窃取 OAuth Code

步骤 5：用 Code 换取 Access Token

真实案例研究：Web 缓存投毒到账户接管 — Joshua Famubode

研究者发现：

  • 通过 X-Forwarded-Host 的全局缓存投毒
  • OAuth 实现信任基于 Cookie 的回调 URL
  • 能够通过 meta 标签覆盖 HttpOnly Cookie
  • 缓存 TTL 足够长以影响多个用户

结果：在缓存窗口期间访问投毒页面的任何用户都会被完全接管账户。

二. 拒绝服务武器化

1：破坏资源加载

赏金范围：$1,000 — $3,000

结果：所有 JavaScript URL 变为 https://target.com:9999/assets/main.js（不可达）
影响：整个应用程序对所有用户崩溃

2：错误页面投毒

赏金范围：$5,000 — $10,000

发生了什么：

  • 缓存键：/api/critical-endpoint
  • 服务器处理：/trigger-500-error
  • 500 错误响应被缓存
  • 所有对该关键端点的 API 调用都失败

真实案例：PayPal 通过缓存投毒 DoS — $9,700 赏金

这是 HackerOne 上薪酬最高的 Web 缓存报告。研究者在 PayPal 的支付处理端点上造成了持久性 DoS。

3：重定向循环投毒

赏金范围：$7,000 — $15,000

对于强制 HTTPS 的应用程序：

结果：

  • 应用程序生成：https://target.com/
  • 浏览器请求：https://target.com/
  • 缓存的响应重定向到：https://target.com/（再次）
  • 无限重定向循环

真实案例：HackerOne DoS 报告 #409370 — X-Forwarded-Port 在 HackerOne.com 本身造成重定向循环。

类似攻击：Exodus 缓存投毒 DoS (#1173153)，使用了 Youstin 的”大规模缓存投毒” 研究中的技术。

三. 信息泄露

1：Session Token 泄露

赏金范围：$3,000 — $8,000

攻击模式：

如果缓存将 .css 视为静态内容：

  • API 返回：{"token": "SECRET123", "email": "victim@example.com"}
  • 缓存将其存储为 CSS 文件
  • 攻击者请求相同 URL
  • 获取受害者的敏感数据

真实案例：Lyst Web 缓存投毒（报告 #631589）

研究者展示了：

“登录用户访问：https://www.lyst.com/shop/trends/mens-dress-shoes/blahblah.css
服务器缓存带有用户会话数据的响应。
未登录的攻击者访问相同 URL 并收到缓存的认证响应。”

2：CSRF Token 泄露

赏金范围：$2,000 — $6,000

如果服务器返回带有 CSRF Token 的 JSON：

并且仅基于路径缓存：所有用户获得相同的 CSRF Token，保护失效。

真实案例：Glassdoor CSRF Token 缓存投毒 (#1424094)

gdToken（反 CSRF Token）在不同页面间被缓存，允许攻击者：

  • 从缓存中收集有效的 CSRF Token
  • 在攻击受害者时使用它们
  • 与 XSS 链接实现完全账户接管

3：管理面板暴露

赏金范围：$8,000 — $20,000

攻击流程：

  • 缓存键：/static/logo.png（被视为安全/公开）
  • 服务器处理：/admin/dashboard（特权页面）
  • 管理面板 HTML 被缓存在公开端点
  • 任何访问 /static/logo.png 的用户都能看到管理面板

为什么是关键的：

  • 暴露敏感的管理功能
  • 可能泄露内部配置
  • 可能揭示其他漏洞
  • 完全绕过认证

真实研究：记录在 James Kettle 的”Practical Web Cache Poisoning” 中，展示了类似的路径混淆攻击。

  • 合法子域名（绕过 CSP）
  • 完全控制载荷
  • 影响所有用户
  • 可以在缓存 TTL 之后持续

2：缓存投毒 → 开放重定向 → OAuth 钓鱼

步骤 1：缓存投毒

步骤 2：OAuth 流程利用
用户发起 OAuth → 被重定向到 evil.com/callback，携带授权码

步骤 3：钓鱼页面

  • URL 以合法域名开头

赏金范围：$8,000 — $15,000

3：缓存欺骗 → 缓存投毒 → 大规模数据泄露

步骤 1：发现缓存欺骗

如果服务器忽略路径后缀并以 CSS 形式返回个人资料数据。

步骤 2：结合缓存投毒

步骤 3：注入数据外带

结果：每个用户的个人资料数据都被外带到攻击者的服务器。

赏金范围：$12,000 — $25,000

真实研究：类似技术记录在 PortSwigger 的”Web Cache Entanglement” 中。

五. 特定框架投毒

1：Next.js 缓存投毒（CVE-2024-46982）

关键漏洞：CVSS 7.5（高）

利用：Next.js 中的 stale-while-revalidate 指令允许攻击者：

  • 投毒 ISR（增量静态再生）缓存
  • 向所有用户提供恶意 JavaScript
  • 即使在缓存重新验证期后仍然持续

真实影响：Next.js 框架漏洞使网站面临缓存投毒

  • 在赏金项目中被利用
  • 在电子商务和金融科技中发现高严重性漏洞
  • 一个实例：存储型 XSS 从主要平台提取敏感用户数据

赏金范围：$8,000 — $20,000（取决于平台范围）

2：WordPress 缓存插件利用

常见问题：

  • W3 Total Cache 信任代理 Header
  • WP Super Cache 不验证 X-Forwarded-Host
  • LiteSpeed Cache 存在路径混淆问题

攻击：将管理 API 响应缓存在公开端点

Django 缓存中间件绕过

利用：

如果缓存在认证之前发生，管理页面会被缓存并带有攻击者的 XSS 载荷。

六. 位置和语言定向攻击

选择性投毒策略

  • 更难检测
  • 绕过自动化扫描器
  • 可以针对特定用户群体
  • 增强隐蔽性

1：语言特定投毒

结果：

  • 英语用户：干净缓存 ✅
  • 西班牙语用户：有毒缓存 ⚠️
  • 德语用户：干净缓存 ✅

防御绕过：安全团队用英语测试，错过了西班牙语缓存。

2：仅移动端投毒

结果：仅移动用户受影响。

  • 独立的移动缓存
  • 移动端通常监控较少
  • 移动端不同的 CSP 策略
  • 移动应用有更大的用户群

3：国家特定投毒

  • 安全监控较少的国家
  • 用户活跃度高的地区
  • 支持有限的国家
  • 地理缓存分区

真实影响：TikTok 广告缓存欺骗（信息泄露） — 地理定向信息泄露赏金 $200。

七. 防护绕过

绕过 WAF 规则

场景：WAF 阻止包含 “evil.com” 的 X-Forwarded-Host

绕过 1：Unicode 规范化

绕过 2：IPv6 表示法

绕过 3：Header 走私

某些后端只验证第一次出现，使用第二次。

绕过 CSP

场景：严格的 CSP 阻止外部脚本

绕过：利用受信任的子域名

如果受信任的子域名存在 XSS，在那里注入载荷：

CSP 允许它，因为它是同源的。

绕过输入验证

场景：应用检查 “http://“ 和 “https://“

绕过 1：协议相对 URL

绕过 2：Data URI

绕过 3：JavaScript URI

十. 真实测试场景

场景 1：电子商务平台

目标：每日 100 万用户的在线商店

攻击路径：

  1. 侦察：

    • 识别：结账流程使用 /checkout 端点
    • 流量：每日约 10 万次结账尝试
    • 缓存：Cloudflare，5 分钟 TTL

  2. 利用：

  1. 影响：
    • 注入假支付表单
    • 窃取信用卡号
    • 每 5 分钟周期影响 347 个用户
    • PCI-DSS 违规

场景 2：SaaS 认证

目标：具有 OAuth SSO 的 B2B SaaS 平台

攻击路径：

  1. 侦察：

    • OAuth 回调：/auth/callback
    • state Cookie 没有 HttpOnly
    • 回调 URL 来自 Cookie 值

  2. 利用链：

  1. 影响：
    • 完整 OAuth 流程劫持
    • 授权码泄露
    • 完全账户接管
    • 多租户数据泄露风险

场景 3：媒体平台

目标：视频流媒体服务

攻击路径：

  1. 侦察：

    • 视频播放器页面：/watch?v=12345
    • 高流量：每日 50 万次视频观看
    • 移动应用共享相同缓存

  2. 利用：

  1. 影响：
    • 专门针对移动用户
    • 通过受信任域名分发恶意软件
    • 广告欺诈注入
    • 订阅者凭据窃取

报告价值：$10,000 — $18,000
