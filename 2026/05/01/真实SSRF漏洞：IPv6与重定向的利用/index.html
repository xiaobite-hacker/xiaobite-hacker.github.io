真实SSRF漏洞：IPv6与重定向的利用

一个来自 HackerOne 的真实 SSRF 漏洞（IPv6 + 重定向）

背景

在现代应用中，Webhook 等功能无处不在。它们实现了自动化、集成和实时工作流。

但它们也引入了危险的东西：

你让服务器代表用户发起请求。

而这正是事情开始出问题的地方。

这是一个看似”安全”的 Webhook 实现如何通过 IPv6 编码和重定向滥用的组合导致 SSRF 的故事。

功能点：Webhook 功能

在 HackerOne 上测试一个目标时，我们发现了一个 Webhook 功能。看起来没什么特别的，直到我们注意到：

  • 它接受用户控制的 URL
  • 它触发服务端请求
  • 它有一些过滤机制

乍一看，它看起来已经加固过了。

应用程序有以下防护：

  • 阻止十进制格式的内部 IP（如 127.0.0.1）
  • 过滤常见的 SSRF 载荷
  • 允许重定向但仍阻止十进制格式的内部 IP

所以……没有直接的 SSRF。

但随后我们聚焦测试一个简单的东西：重定向。

我们发现后端自动跟随重定向，但不会重新验证目标地址。这是一个经典错误。因此，我们没有直接针对内部资源，而是尝试：

http://attacker.com → 重定向 → 内部资源（十进制）→ 仍然被阻止。

所以我们深入探索。

Webhook 响应极其不一致……到处是延迟，没有可靠的时间信号，没有清晰的信号。开放端口、关闭端口……看起来都一样。自动化工具毫无用处。

所以我们调整了策略。

我们不再依赖工具，而是改为手动——缓慢、专注、有目的性地逐个探测知名端口，寻找任何行为不同的东西。

关键技术要点总结

1. SSRF 绕过的核心思路

当目标有 SSRF 防护时，不要直接尝试 127.0.0.1，考虑：

  • IPv6 映射地址：[::ffff:7f00:1] 等价于 127.0.0.1
  • 八进制表示：0177.0.0.1 等价于 127.0.0.1
  • 十六进制表示：0x7f.0x0.0x0.0x1
  • 十进制整数：2130706433 等价于 127.0.0.1

2. 重定向利用模式

许多 SSRF 过滤器在初始请求时验证 URL，但不会在跟随重定向后重新验证。

3. Webhook 是高价值 SSRF 入口

Webhook 功能的本质就是让服务器发起请求，天然适合 SSRF。测试时重点关注：

  • 是否接受用户控制的 URL
  • 重定向是否被跟随
  • 重定向后是否重新验证

4. 手动测试的重要性

当自动化工具因响应不一致而失效时，手动逐端口探测仍然有效。耐心和专注比工具更重要。
